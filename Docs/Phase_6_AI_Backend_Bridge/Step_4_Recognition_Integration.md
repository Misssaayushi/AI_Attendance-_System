# Step 4: Recognition Loop Integration

## Objective
Integrate the API service into `ai_module/recognize_faces.py` so verified attendance events are transmitted to the backend or mock service.

## Integration Rule
Transmission must happen only after Phase 5 verification succeeds:

1. Face is recognized.
2. Confidence threshold passes.
3. Stability streak passes.
4. Cooldown check passes.
5. API event is transmitted.

No API call should happen for unknown faces, low-confidence faces, unstable faces, or cooldown-active students.

## Tasks
1. Import `AttendanceAPIService`.

2. Initialize the API service once before the webcam loop starts.
   - Do not create a new HTTP client for every frame.

3. Trigger transmission only when:
   - `verified=True`
   - `student_id` is valid
   - `name` is not `UNKNOWN_LABEL`

4. Send AI-owned event data.
   - `student_id`
   - `name`
   - `confidence`
   - current timestamp generated by the AI module
   - `status="Present"`

5. Store API feedback per student or per visible face.
   - Success message: `Attendance Logged`
   - Server failure: `Server Error`
   - Timeout: `Backend Timeout`
   - Connection failure: `Backend Offline`
   - Mock mode: `Mock Success` or `Mock Failure`

6. Display API feedback in the OpenCV overlay.
   - Keep Phase 5 status visible while waiting for verification.
   - Replace or append server feedback after transmission.
   - Use color coding:
     - Green for logged attendance.
     - Yellow for pending or mock feedback.
     - Red for backend failure.

7. Preserve real-time stability.
   - Never crash the loop on API errors.
   - Keep timeout and retry values small.
   - Avoid duplicate calls by relying on Phase 5 cooldown.

## Expected Outcome
`recognize_faces.py` becomes the complete AI-side workflow:
Recognition -> Verification -> Transmission -> Overlay Feedback.

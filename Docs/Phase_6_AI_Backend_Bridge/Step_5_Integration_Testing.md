# Step 5: Integration Testing

## Objective
Create `ai_module/tests/test_integration.py` to verify the Phase 6 communication bridge without implementing backend database logic.

## Test Scope
These tests validate AI-side communication behavior only. They do not test MySQL, FastAPI route internals, React dashboards, or database persistence.

## Tasks
1. Test payload creation.
   - Required fields are present.
   - Timestamp is generated by the AI module.
   - Confidence remains numeric.
   - Status defaults to `Present`.

2. Test mock success mode.
   - Enable mock mode.
   - Send a sample verified attendance event.
   - Confirm response success is true.
   - Confirm no real backend is required.

3. Test mock failure mode.
   - Enable mock mode with forced failure.
   - Confirm response success is false.
   - Confirm the returned message is display-friendly.

4. Test successful backend response simulation.
   - Simulate `200 OK`.
   - Simulate `201 Created`.
   - Confirm both are treated as successful attendance logging.

5. Test failed backend response simulation.
   - Simulate `400 Bad Request`.
   - Simulate `401 Unauthorized`.
   - Simulate `500 Internal Server Error`.
   - Confirm failures return safe error messages.

6. Test network resilience.
   - Simulate connection error.
   - Simulate timeout.
   - Confirm exceptions do not escape the API service.

7. Test duplicate-transmission safety at integration level.
   - Verify that `recognize_faces.py` only calls the API service after `AttendanceManager.verify_attendance()` returns `verified=True`.
   - Confirm cooldown-active results do not trigger new transmissions.

## Suggested Testing Approach
Use lightweight mocking around the `requests.post` call instead of starting a real backend. A tiny local mock server may be added only if manual testing is needed.

## Expected Outcome
The tests prove that Phase 6 can safely transmit attendance events, survive backend failures, and preserve Phase 5 zero-duplicate behavior.
